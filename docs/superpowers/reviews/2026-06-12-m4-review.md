# M4 filing loop — multi-agent review (2026-06-12)

Three review agents (security, architecture, correctness) audited the whole M4 filing
loop (backend `naming/encode/tagging/library/filing/actions/settings/ipc_filing` + db v4,
frontend `filing.ts/sift-live.ts/report-view.ts`). This records what was fixed and what is
deliberately deferred (with the rationale), so the deferrals are tracked, not lost.

## Fixed (committed)

- **Path-traversal containment** (security HIGH). `create_bin`/`file_track` joined an
  untrusted `bin_rel`/`parent_rel` onto the root; `..` or an absolute/drive path escaped it.
  Added `library::safe_join` (sanitizes every segment, refuses `..`/absolute/`\`); all bin
  destinations funnel through it. Tests added. → commit "path-traversal containment guard".
- **Undo consistency** (correctness HIGH / arch HIGH). `revert_batch` now also clears
  `target_format`/`confidence` and deletes the filing-time `metadata` row (keeps
  `analyzed_at` — file unchanged, verdict still valid). Documented that the embedded tag
  write is not reversed. Test asserts the cleared columns.
- **Filing atomicity** (correctness HIGH). If journaling fails *after* the move/encode,
  `file_track` now rolls back (rename the file back / delete the orphan transcode) so a DB
  error can't leave a lost or orphaned file.
- **Drag-drop zone targeting** (correctness MED). Removed the `devicePixelRatio` division in
  `dropModeAt` — Tauri 2 already emits logical pixels, so it double-corrected on HiDPI/
  scaled Windows and mis-targeted the zone.
- **Stale detail pane after undo** (correctness MED). The toast "Annuler" now clears `#mid`.
- **db v4 comment** lists `wav_16_44` (cosmetic).

## Deferred — need the user / runtime, or a product decision

- **DB lock held across the ffmpeg encode** (arch HIGH — the agent's #1). `ipc_filing`
  holds the single `Mutex<Connection>` for the whole `file_track`, so the multi-second
  encode serializes the analysis workers AND every other IPC/UI refresh → the window
  freezes during a transcode. Worst case is `file_batch` (not yet wired to the UI); the
  current single-track Ranger freeze is bounded to one encode. The clean fix (split
  `file_track` into locked-prepare → lock-free-encode → locked-commit) is invasive on the
  core happy-path and the just-added compensation ordering; **worth doing, but with the app
  running to verify** — not blind. Highest-priority next.
- **`naming.rs` JUNK_TOKENS false positives** (arch MED). Bare substrings `320/256/192/128/
  rip/_/[ /http` match real titles ("Apollo 128", "Rip It Up", underscores). This is the
  green/yellow gate, so changing it is a product-behavior decision + test churn — wants the
  user's call on what counts as junk. Recommend word-boundary anchoring and treating `_` as
  a normalization, not a disqualifier.
- **`file_batch` undoes per-track, not atomically** (arch MED). Each track gets its own
  `batch_id`, so a batch import is N undo units, contradicting the "one action = one batch"
  docstring. Not user-visible until batch filing is wired. Give it an umbrella batch id then.
- **"+ nouveau" always creates at root** (`filing.ts makeBin`, correctness MED). Ignores the
  selected tree node; drilling into House/Deep and creating gives a root-level bin. Needs
  nested-creation UX (pass the selected rel as parent).
- **`is_conformant(Mp3320)` ignores bitrate** (correctness MED). Any `.mp3` is "conformant"
  and moved without re-encode (deliberate, to avoid generation loss) but the stored
  `target_format='mp3_320'` then mislabels a 128 kbps file. Naming/contract mismatch, not a
  bug — decide whether to honor the configurable bitrate threshold here.
- **`#pa` dual click listeners** (arch HIGH, but no live bug). app.js + sift-live both listen
  on `#pa`; works only because attributes don't collide and is undocumented contract.
  `stopImmediatePropagation` wouldn't fix it (app.js listens first). Track when refactoring.
- **ffmpeg path hardening / import_paths canonicalization** (security MED/LOW). Low real risk
  for a local single-user tool; a naive protocol/`-` guard would break Windows `C:\` paths,
  so deferred deliberately.
- **module-level `#![allow(dead_code)]`** now that M4 is wired (arch LOW) — removing them
  would surface genuinely-unused items (e.g. `TRASH_PURGE_DAYS`); cleanup, not urgent.
